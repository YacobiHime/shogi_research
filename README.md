# 将棋エンジンのUSIオプション最適化

## CMA-ESとランダムサーチの比較

## 1. 概要

本研究では、将棋エンジン「やねうら王」の評価関数や探索アルゴリズムを変更せず、USIプロトコルを通じて外部から変更できるオプションを自動調整する。

最適化手法には **CMA-ES（Covariance Matrix Adaptation Evolution Strategy）** を使用し、デフォルト設定を基準とした自己対局の平均得点を目的関数として、より良いオプション設定を探索する。

さらに、同一の候補評価回数および総対局数の下でランダムサーチを実施し、次の点を比較する。

1. デフォルト設定より高い対局成績を持つ設定を発見できるか
2. CMA-ESはランダムサーチより少ない評価回数で良い設定を発見できるか
3. 少数対局による評価ノイズが最適化結果にどの程度影響するか
4. 探索中に得られた改善が、独立した検証対局でも再現されるか

---

## 2. 研究テーマ

> **将棋エンジンのUSIオプションをCMA-ESで自動調整する**
>
> **― ランダムサーチとの探索効率比較 ―**

---

## 3. 研究背景

将棋エンジンには、評価関数や探索アルゴリズム以外にも、定跡の選択方法や引き分け評価など、対局時の挙動を制御する複数のオプションが存在する。

これらのオプションは手動で設定されることが多いが、複数のオプションを同時に調整する場合、組合せの数が増加し、人手による網羅的な探索は困難になる。

そこで本研究では、ブラックボックス最適化手法であるCMA-ESを使用し、対局結果だけを利用してUSIオプションを自動調整する。

ただし、将棋エンジンの対局結果には次のようなノイズが含まれる。

* 開始局面による有利不利
* 先手・後手による差
* 短時間対局における探索の不安定性
* 並列実行や時間制御による探索量の変動
* 少数対局から計算した勝率の標本誤差

そのため、本研究では単に探索中の最高勝率を報告するのではなく、探索用対局と検証用対局を分離し、独立した対局によって最終設定の再現性を評価する。

---

## 4. 研究目的

### 4.1 主目的

限られた対局予算の下で、CMA-ESがランダムサーチより効率的に高得点のUSIオプション設定を発見できるかを検証する。

### 4.2 副目的

以下についても検証する。

* USIオプションの変更によって、デフォルト設定に対する対局成績を改善できるか
* 少数対局で測定した候補の得点が、十分な対局数による再評価でも維持されるか
* CMA-ESとランダムサーチで、探索結果のばらつきに違いがあるか
* 発見された設定が特定の開始局面や乱数シードに過度に依存していないか
* パラメータ間に相互作用が存在するか

---

## 5. 研究課題

本研究では、次の研究課題を設定する。

### RQ1：デフォルト設定からの改善

USIオプションを最適化することで、デフォルト設定に対する平均得点を改善できるか。

### RQ2：探索効率

同一の候補評価回数および総対局数において、CMA-ESはランダムサーチより高得点の設定を発見できるか。

### RQ3：結果の再現性

探索中に高い得点を示した設定は、探索に使用していない開始局面および乱数シードによる検証対局でも高い得点を維持するか。

### RQ4：評価ノイズの影響

候補当たりの対局数を変化させた場合、最適化結果の安定性はどのように変化するか。

---

## 6. 仮説

### H1

CMA-ESまたはランダムサーチによって、デフォルト設定より高い検証得点を持つ設定を発見できる。

### H2

同一の対局予算では、CMA-ESの最良検証得点はランダムサーチの最良検証得点を上回る。

### H3

候補当たりの対局数が少ない場合、探索中の最高得点と独立検証時の得点の差が大きくなる。

## 7. 研究範囲

本研究で変更するのは、使用するやねうら王の実行ファイルがUSIオプションとして公開している設定のみとする。

以下は変更しない。

* NNUE評価関数の重み
* 探索アルゴリズムのソースコード
* 探索パラメータの内部定数
* 定跡データベースの内容
* エンジンのソースコード
* コンパイルオプション
* 使用するCPUスレッド数
* ハッシュサイズ
* 対局時間または探索ノード数

したがって、本研究が評価するのは「やねうら王全体を最適化する手法」ではなく、**特定のエンジン・評価関数・定跡ファイル・対局条件における、公開USIオプションの外部最適化**である。

## 9. 最適化対象オプションの選定

### 9.1 暫定候補

当初の候補は以下のとおりとする。

| オプション                   |   型 | 想定される役割          | 注意点                 |
| ----------------------- | --: | ---------------- | ------------------- |
| `Contempt`              |  整数 | 引き分け局面の評価に関係する設定 | バージョンによっては存在しない     |
| `BookEvalDiff`          |  整数 | 定跡候補として許容する評価値差  | 定跡を使用する場合のみ有効       |
| `BookDepthLimit`        |  整数 | 定跡候補のdepthに関する下限 | 定跡オプション方式によって名称が異なる |
| `ConsiderBookMoveCount` | 真偽値 | 定跡の採択数を考慮した確率的選択 | 連続変数ではない            |

### 9.2 バージョン依存性

`BookDepthLimit` は、使用する定跡オプション方式によって、次のような別オプションへ置き換えられる場合がある。

* `BookDepthBlackLimit`
* `BookDepthWhiteLimit`

また、`Contempt` は使用するバージョンに存在しない可能性がある。

このため、最終的な探索対象は実験開始前に取得した `usi_options.txt` に基づいて確定する。

### 9.3 採用条件

以下のすべてを満たすオプションのみを最適化対象とする。

1. 使用する実行ファイルのUSI出力に存在する
2. `setoption` によって外部から変更できる
3. 設定変更が正常に反映される
4. 評価関数や探索コードの再コンパイルを必要としない
5. 実験条件下で実際に対局挙動へ影響する
6. 許容範囲と型を機械的に取得または明示できる

### 9.4 事前感度確認

本実験の前に、各オプションについて次の値を試す。

* デフォルト値
* 探索範囲の下側
* 探索範囲の上側

同一開始局面・同一条件で対局または思考ログを比較し、設定変更が実際の指し手や定跡選択に影響するかを確認する。

影響が確認できないオプションは、探索対象から除外する。

## 10. パラメータ表現

候補設定を次のベクトルとして表す。

[
\mathbf{x}=(x_1,x_2,\ldots,x_d)
]

ここで (d) は採用されたUSIオプションの数である。

### 10.1 連続値からUSI値への変換

CMA-ESは連続空間を前提とするため、USIオプションの型に応じて変換する。

#### 整数型

CMA-ESが生成した値を丸めて整数に変換する。

[
v_i=\operatorname{round}(x_i)
]

#### 真偽値型

0.5を閾値として変換する。

[
v_i=
\begin{cases}
\text{false} & x_i < 0.5\
\text{true} & x_i \geq 0.5
\end{cases}
]

ただし、真偽値をCMA-ESで扱うことには不連続性がある。

そのため、次の2種類の実験を区別する。

* **主実験：整数型オプションのみ**
* **追加実験：真偽値オプションを含む混合変数最適化**

これにより、CMA-ESに有利または不利な変数表現が結論へ混入することを防ぐ。

### 10.2 正規化

各パラメータは探索範囲が異なるため、最適化器内部では原則として ([0,1]) に正規化する。

[
z_i=\frac{v_i-L_i}{U_i-L_i}
]

* (L_i)：下限
* (U_i)：上限

CMA-ESは正規化空間上で実行し、評価時にUSI値へ逆変換する。

## 11. 探索範囲

探索範囲は、USI出力に表示された機械的な最小値・最大値をそのまま使用するとは限らない。

非常に広い範囲をそのまま探索すると、実用上意味のない候補が多数生成される可能性があるため、次の手順で範囲を決定する。

1. USI出力から正式な許容範囲を取得する
2. 公式説明とデフォルト値を確認する
3. 予備実験によって挙動が変化する範囲を調べる
4. 本実験で使用する探索範囲を確定する
5. 範囲決定の理由を記録する

最終的な範囲は次の表に記入する。

| オプション |  型 | デフォルト値 | USI上の範囲 | 実験上の探索範囲 | 採用理由 |
| ----- | -: | -----: | ------: | -------: | ---- |
| 未確定   |    |        |         |          |      |

### 境界処理

単純なクリッピングは、境界値へ候補が集中し、CMA-ESの分布更新を歪める可能性がある。

そのため、優先順位を次のようにする。

1. pycmaの境界制約機能を使用する
2. 範囲内へ変換する写像を使用する
3. 範囲外候補を再生成する
4. 単純クリッピングは最後の手段とする

### 13.1 基本条件

| 項目      | 条件              |
| ------- | --------------- |
| 候補エンジン  | 最適化対象の設定        |
| 基準エンジン  | デフォルト設定         |
| 実行ファイル  | 両者で同一           |
| 評価関数    | 両者で同一           |
| 定跡ファイル  | 両者で同一           |
| スレッド数   | 両者で同一           |
| ハッシュサイズ | 両者で同一           |
| 投了値     | 両者で同一           |
| 入玉ルール   | 両者で同一           |
| 千日手規則   | 両者で同一           |
| 開始局面    | 固定された開始局面集合から選択 |
| 手番      | 同一局面を先後入れ替えて対局  |

### 13.2 探索量の固定

可能であれば、1手当たりの秒数ではなく、1手当たりの探索ノード数を固定する。

固定ノード数を優先する理由は、時間制御では以下の影響を受けるためである。

* CPU負荷
* OSのスケジューリング
* 同時実行数
* エンジン初期化時間
* 短時間制御における時間計測誤差

ただし、使用する対局管理機能でノード数固定が困難な場合は時間制御を使用し、その理由と実測探索ノード数を記録する。

時間制御を使用する場合、`1手0.1秒` を確定条件とはせず、予備実験で以下を確認する。

* 正常に十分な深さまで探索できるか
* タイムアウトが発生しないか
* 同一条件で探索ノード数が極端に変動しないか
* 定跡を抜けた後の対局が不自然にならないか

## 14. 開始局面

### 14.1 開始局面を固定する理由

初期局面から繰り返し対局すると、定跡選択の偏りや同一対局の重複が生じる可能性がある。

そこで、複数の開始局面からなる開始局面集を作成する。

### 14.2 条件

開始局面集は以下を満たすようにする。

* 合法局面である
* 一方に明らかな勝勢がない
* 特定の戦型だけに偏らない
* 同一局面を先後入れ替えて使用できる
* 探索用と検証用で重複しない
* SFEN形式で保存する
* 生成方法と乱数シードを記録する

### 14.3 定跡オプションとの整合性

定跡関連オプションを最適化する場合、開始局面が定跡ファイルに登録されているかを確認する。

開始局面がすべて定跡範囲外の場合、定跡オプションを変更しても対局へ影響しない可能性がある。

そのため、次のいずれかを採用する。

* 初期局面または浅い開始局面から対局する
* 使用する開始局面が定跡データベースに存在することを確認する
* 定跡が実際に使用されたかをログへ記録する

## 15. 対局得点

候補設定 (\mathbf{x}) の成績を次の平均得点で表す。

[
S(\mathbf{x})=
\frac{
N_{\mathrm{win}}+
0.5N_{\mathrm{draw}}
}{
N_{\mathrm{games}}
}
]

ここで、

* (N_{\mathrm{win}})：候補設定の勝利数
* (N_{\mathrm{draw}})：引き分け数
* (N_{\mathrm{games}})：総対局数

である。

CMA-ESの実装が最小化を前提とする場合、目的関数は次のようにする。

[
f(\mathbf{x})=-S(\mathbf{x})
]

または、

[
f(\mathbf{x})=1-S(\mathbf{x})
]

READMEおよび実装内では、最大化する「得点」と最小化する「目的関数」を混同しないよう区別する。

## 16. 対局のペアリング

同一の開始局面について、候補設定と基準設定の先後を入れ替えた2局を実施する。

開始局面 (p_i) に対して、次のペアを作る。

1. 候補設定が先手、デフォルト設定が後手
2. デフォルト設定が先手、候補設定が後手

この2局を1つの対局ペアとする。

候補当たり20局を行う場合は、10局面について先後入れ替えを行う。

ただし、20局から得られる得点はノイズが大きいため、20局の結果だけで最終的な強さを判断しない。

## 17. 実験の段階

本研究は、次の3段階で実施する。

### Phase 0：動作確認

目的：

* USIオプションが正常に反映されるか確認する
* 対局プログラムのバグを除去する
* 定跡が実際に使用されているか確認する
* 対局結果の保存形式を確認する

この段階の対局結果は、本実験の統計には含めない。

### Phase 1：予備実験

目的：

* 探索範囲を決定する
* 候補当たりの対局数を検討する
* 適切な時間またはノード数を決定する
* パラメータ変更による効果の有無を確認する
* CMA-ESの初期標準偏差を決定する

### Phase 2：最適化実験

CMA-ESとランダムサーチを同一予算で実行する。

### Phase 3：独立検証

各探索手法が発見した最良候補を、探索に使用していない開始局面と乱数シードで再評価する。

## 18. CMA-ES

### 18.1 基本設定

暫定的に次の設定を使用する。

| 項目     |                暫定値 |
| ------ | -----------------: |
| 次元数    |           採用オプション数 |
| 個体数    | pycmaのデフォルト値または6～8 |
| 世代数    |          総評価予算から決定 |
| 初期平均   |            デフォルト設定 |
| 初期標準偏差 |            予備実験で決定 |
| 境界     |     正規化空間の ([0,1]) |
| 乱数シード  |            実行ごとに記録 |

個体数や世代数を先に固定するのではなく、総候補評価数を比較予算として決める。

例えば、

[
B=120
]

を候補評価予算とした場合、CMA-ESとランダムサーチはそれぞれ120候補を評価する。

### 18.2 重複候補

整数化や真偽値化によって、異なる連続値が同一のUSI設定へ変換される場合がある。

そのため、次を記録する。

* CMA-ESが生成した連続ベクトル
* USI値へ変換した候補
* 同一候補の過去評価回数
* 重複候補を再評価したか
* キャッシュを利用したか

原則として、評価ノイズを考慮する場合は重複候補を単純に除外せず、再評価として扱う。

---

## 19. ランダムサーチ

ランダムサーチでは、CMA-ESと同一の探索範囲から候補を生成する。

比較の公平性を保つため、以下を一致させる。

* 評価する候補数
* 候補当たりの対局数
* 総対局数
* 使用する開始局面集合
* 先後の割り当て
* エンジン設定
* ハードウェア
* 並列実行数
* 検証方法

整数変数は探索範囲内の整数から一様に生成する。

真偽値変数は原則として等確率で生成する。

CMA-ESと同様に、各候補の連続表現、USI値、得点および対局結果を保存する。

---

## 20. 共通乱数と評価局面

CMA-ESとランダムサーチの比較では、同じ評価番号の候補に対して、可能な限り同一の開始局面ペアを使用する。

これにより、開始局面の難易度差によるばらつきを小さくする。

ただし、すべての候補を常に同じ少数局面だけで評価すると、その局面集合への過適合が起こる。

そのため、次のいずれかを採用する。

### 方法A：評価番号ごとの固定局面

評価番号 (j) ごとに開始局面集合 (P_j) を事前生成し、両手法で共通利用する。

### 方法B：複数の局面バッチを循環利用

複数の開始局面バッチを用意し、候補評価ごとに循環させる。

いずれの場合も、探索用局面と最終検証用局面は完全に分離する。

---

## 21. 探索予算

比較の単位は世代数ではなく、次の総予算とする。

[
G_{\mathrm{total}}
==================

N_{\mathrm{candidates}}
\times
N_{\mathrm{games\ per\ candidate}}
]

例えば、

* 候補数：120
* 候補当たり：20局

の場合、

[
G_{\mathrm{total}}=2400
]

局となる。

CMA-ESとランダムサーチでこの総対局数を一致させる。

最終検証対局は探索予算とは別に計上する。

---

## 22. 反復実験

1回の探索結果だけでは、最適化器の乱数や対局ノイズの影響を分離できない。

そのため、可能であれば各手法を複数回実行する。

### 推奨

* CMA-ES：5回以上
* ランダムサーチ：5回以上

各反復で以下を変更する。

* 最適化器の乱数シード
* 候補生成の乱数シード
* 開始局面の割り当て

一方、同じ反復番号のCMA-ESとランダムサーチでは、評価局面の条件をそろえる。

計算資源が不足する場合でも、最低3反復を目標とし、1反復しか実施できない場合は探索的実験として扱う。

---

## 23. 最終候補の選択

探索中の最高得点候補をそのまま最終結果とはしない。

各手法について、探索中の上位 (k) 候補を選び、追加対局によって再評価する。

暫定的には、

[
k=3
]

とする。

選択手順は次のとおりである。

1. 探索中の得点で上位3候補を選ぶ
2. 探索用とは別の開始局面で追加対局する
3. 追加対局の得点が最も高い候補を、その手法の代表候補とする
4. 代表候補を最終検証へ進める

これにより、少数対局で偶然高得点になった候補を選択する危険を減らす。

---

## 24. 独立検証

### 24.1 対象

以下を独立検証する。

* デフォルト設定
* CMA-ESによる代表候補
* ランダムサーチによる代表候補

### 24.2 条件

検証には以下を使用する。

* 探索に使用していない開始局面
* 探索に使用していない乱数シード
* 十分な対局数
* 先後入れ替え
* 探索時と同一のエンジン・評価関数・定跡ファイル

### 24.3 対局数

20局は候補を粗く評価するためには利用できるが、最終的な強さの判定には少なすぎる可能性が高い。

最終検証では、計算資源に応じて次を目標とする。

| 検証段階     |         推奨対局数 |
| -------- | ------------: |
| 上位候補の再選抜 | 候補当たり100～200局 |
| 最終比較     |  組合せ当たり400局以上 |
| 高精度な差の推定 |    1000局以上を検討 |

最終的な対局数は事前に決め、結果を見てから増減させない。

---

## 25. 統計評価

### 25.1 報告する値

各設定について、最低限以下を報告する。

* 勝数
* 敗数
* 引き分け数
* 平均得点
* 得点差
* 95%信頼区間
* 総対局数

### 25.2 主比較

主比較は、独立検証における次の差とする。

[
\Delta_{\mathrm{CMA}}
=====================

S_{\mathrm{CMA}}-S_{\mathrm{default}}
]

[
\Delta_{\mathrm{random}}
========================

S_{\mathrm{random}}-S_{\mathrm{default}}
]

さらに、

[
\Delta_{\mathrm{method}}
========================

S_{\mathrm{CMA}}-S_{\mathrm{random}}
]

を比較する。

### 25.3 探索効率

候補評価回数 (t) までに発見された最良の再評価得点を、

[
B(t)
]

として記録する。

CMA-ESとランダムサーチの以下を比較する。

* Best-so-far曲線
* 曲線下面積
* 最終的な最良検証得点
* 所定の得点へ到達するまでの評価回数
* 反復間のばらつき

### 25.4 多重比較

多数の候補を同時に評価すると、偶然高得点になった候補が選ばれやすい。

したがって、探索中の最大得点について統計的有意性を主張せず、独立検証結果を主な結論に使用する。

## 26. 実験手順

1. 使用するエンジン、評価関数、定跡ファイルを固定する
2. 各ファイルのバージョンとハッシュを記録する
3. `usi` コマンドの出力を保存する
4. 利用可能なUSIオプションと型を取得する
5. 各オプションの動作確認を行う
6. 最適化対象と探索範囲を確定する
7. 探索用・再選抜用・最終検証用の開始局面を分割する
8. デフォルト設定同士の対局を行い、先手差や局面差を確認する
9. CMA-ESを規定予算で実行する
10. ランダムサーチを同一予算で実行する
11. 各手法の上位候補を追加対局で再評価する
12. 各手法の代表候補を決定する
13. 未使用局面による最終検証を実施する
14. 信頼区間と探索効率を計算する
15. 複数反復の結果を集計する
16. 実験設定、乱数シード、棋譜、集計結果を公開する

## 30. 考察項目

結果は以下の観点から分析する。

### 30.1 探索効率

* CMA-ESはランダムサーチより早く高得点候補へ到達したか
* 最終的な代表候補の性能に差があったか
* 反復ごとの結果は安定していたか

### 30.2 評価ノイズ

* 探索時得点と検証時得点にどの程度の差があったか
* 20局評価で候補順位は安定したか
* 候補当たりの対局数を増やすと順位相関は改善したか

### 30.3 過適合

* 探索用局面での改善が未使用局面でも再現されたか
* 特定の戦型や開始局面だけで得点していないか
* デフォルト設定だけを相手にしたことによる相手依存性がないか

### 30.4 パラメータの解釈

* 最適値が探索範囲の境界に集中したか
* パラメータを単独変更した場合にも効果があるか
* 複数パラメータの組合せによってのみ改善したか
* 定跡利用率や引き分け率にどのような変化があったか

### 30.5 混合変数の影響

* 真偽値オプションを含めることでCMA-ESの効率が低下したか
* 整数型のみの場合と結論が変化したか

## 31. 本研究で主張できる範囲

本研究の結果から直接主張できるのは、次の範囲に限定される。

> 固定したやねうら王のバージョン、評価関数、定跡ファイル、ハードウェア、開始局面集合および対局条件において、CMA-ESまたはランダムサーチがデフォルト設定より高い対局得点を持つUSIオプション設定を発見できたか。

次のような一般化は、追加実験なしには主張しない。

* 発見した設定があらゆる持ち時間で強い
* 他の評価関数でも同様に強い
* 他の定跡ファイルでも同様に強い
* 他のやねうら王バージョンでも強い
* 他の対戦相手に対しても強い
* CMA-ESが一般に将棋エンジン調整で最良の手法である
* 発見した設定が人間の専門家による調整より優れている

人手調整との比較を主張する場合は、別途、調整者・調整時間・試行回数を管理した人手調整群を用意する必要がある。

## 32. 想定される限界

* 対局結果が確率的であり、候補の真の性能を直接観測できない
* 最適化対象が少数のUSIオプションに限定される
* 真偽値や整数値はCMA-ESが本来想定する連続空間と異なる
* デフォルト設定だけを対戦相手にすると、相手固有の弱点へ適応する可能性がある
* 短時間または少ノード対局で得られた設定が長時間対局でも有効とは限らない
* 定跡関連オプションの効果は、使用する定跡ファイルと開始局面に強く依存する
* 並列実行時にはCPU資源の競合が発生する可能性がある
* 計算予算によっては十分な反復数や検証対局数を確保できない

## 35. 現時点での重要な判断

本研究では、次の方針を採用する。

1. 研究目的を「人手調整との比較」ではなく「CMA-ESとランダムサーチの比較」とする
2. 使用する実行ファイルのUSI出力を確認してから最適化対象を確定する
3. 主実験では整数型オプションを優先する
4. 真偽値オプションを含む実験は追加実験として分離する
5. 20局評価は探索用の近似評価として扱う
6. 最終結果は独立した多数対局によって決定する
7. 探索用・候補選択用・最終検証用の開始局面を分離する
8. CMA-ESとランダムサーチでは総候補数と総対局数を一致させる
9. 単一実行ではなく、可能な限り複数の乱数シードで反復する
10. 世代数ではなく累積候補評価数または累積対局数で探索効率を比較する
